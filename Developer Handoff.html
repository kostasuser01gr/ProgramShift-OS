Next.js · Google Sheets · Vercel

The Vercel app, hardened

A verified Next.js application that reads and writes your live spreadsheet through a service account, coalesces reads in a short-lived in-memory cache, and stays fresh via an Apps Script webhook. Staff data is authenticated, manager writes are validated, and the primary schedule flows are implemented.

00What this is (and isn't)

Is: the unified production data path — Google or credentials authentication, protected OS assets, environment-driven role gating, authenticated schedule hydration, the Sheets repository, read cache, validated manager writes with UI rollback, and the signed webhook.

Current boundary: schedule reading and manager cell editing are live. Requests, reservations, chat, tasks, notes, files, forms, publication settings, and administration tools are explicitly labelled browser-local until server persistence is added.

The one principle

Everything touches Google through lib/sheets.ts (the SheetsRepository). In phase 3 you replace that one file with a PostgresRepository of the same shape — app/ and compute.ts never change.

01Data flow

Google / credentials session→protected OS→GET /api/schedule→memory cache→Sheets API
Manager edits→POST /api/cell→writes ΩΡΑΡΙΑ→invalidate cache
Sheet edited directly→Apps Script onEdit→POST /api/webhook/sheets→invalidate cache

Reads are cheap (cache, 60s TTL); writes go to the human-readable sheet only — the bound script mirrors them into CODED and runs validation. Two directions, two mechanisms.

03Setup

  1. Install

    cd vercel-app && npm install

  2. Service account

    Google Cloud → create a service account, enable the Sheets API, download the JSON key, and share the spreadsheet with the service-account email (Editor — the app writes cells).

  3. OAuth client

    Create a Google OAuth Web client; redirect URI http://localhost:3000/api/auth/callback/google (+ prod).

  4. Env

    Copy .env.example → .env.local and fill it in (next section).

  5. Run

    npm run dev → open localhost:3000. Reads use a short-lived per-instance cache.

  6. Verify

    npm run lint && npm run typecheck && npm test && npm run build && npm run audit:prod.

05Webhook glue

Already wired into the delivered Code.gs (notifyApp_() fires from the installable onEdit). Configure it in Apps Script → Project Settings → Script Properties:

APP_WEBHOOK_URL    = https://your-app.vercel.app/api/webhook/sheets
APP_WEBHOOK_SECRET = <same string as WEBHOOK_SECRET in Vercel>

The handler verifies the X-Secret header and busts the cache — so a direct sheet edit shows up in the app within one read.

08Suggested next steps

In order of value:

1 · Requests modelAdd a persistent requests store so swaps and leave requests drive manager decisions and notifications.
2 · Shared cacheMove from per-instance memory to a shared cache when traffic requires cross-instance invalidation.
3 · RealtimeHave the webhook push to clients via SSE so edits appear without a refresh.
4 · PWA + i18nAdd the manifest/service worker and next-intl (EL/EN) for the employee app.
