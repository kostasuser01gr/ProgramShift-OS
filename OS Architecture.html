Modular operating platform

Program Shift OS

The shift schedule became a platform. One shell, one app dock, a command palette, a global AI assistant, automations that act on the live data, notifications, an audit trail, and 17 modules — all sharing a single services layer, all role-aware, in Greek & English, light & dark.

00What this is

A genuinely working front-end operating platform built around your June schedule. Every visible feature either works against the live in-memory data or is clearly labelled Coming soon. It runs entirely in the browser — no build step — and persists tasks, notes, chat, automations and settings in localStorage.

Design principle

Nothing is faked. The launcher opens real modules; the command palette runs real commands; automations read the real coverage/warnings and fire real notifications + audit entries; the AI sees the real month. What needs a server (file storage, real multi-user sync) is marked, not stubbed.

05AI layer

The assistant assembles a context object — current month, role, open app, understaffed days, warning counts — and asks the live model; if none is reachable it answers from computed analytics, so it’s never dead. Every query is audited. ChatOps in the chat module (/coverage, /warnings) and the Gemini gem link round out the AI surface.

Question→build context (month, role, derived)→live model→answer
fallback if offline→computed answer from coverage/warnings/fairness

07Run & extend

Run: open Program Shift OS.html — no build, no install. Press ⌘K for the command palette.

Add a module in 2 steps

1 · Write a component that calls useOS() for state. 2 · Add a row to REGISTRY in shell.jsx and register it in COMP. It appears in the launcher, dock, palette and settings automatically, gated by the permission you name.

09Honest limits

  • Single-user, single-browser: stores persist locally, not across devices (that’s the Vercel phase).
  • Files & Form Builder are labelled Coming soon — they need server storage.
  • The AI uses a live model when the host exposes one, else a computed fallback; it isn’t a fine-tuned scheduler yet.
  • Offline/queue, rate-limiting and session handling are production concerns that live in the Next.js layer, not the prototype.
  • In-browser Babel prints one performance warning — expected for a no-build prototype; precompile for production.
Next recommended

Auto-rota generation + “Fix this” on each warning (turn the validator into a scheduler), then port the platform onto the Vercel scaffold so the stores become real and multi-user.
