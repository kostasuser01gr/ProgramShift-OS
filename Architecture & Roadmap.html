Vercel · Google Workspace · Scale

From a guarded spreadsheet
to an app everyone can use

You don't have to abandon the Sheet to get a real web/mobile app. The cleanest path keeps Google as the system of record and puts a Vercel front-end on top — then migrates the data layer only when scale actually demands it. Here's how everything connects, and the order to build it.

00The short answer

Recommended shape

Build a Next.js app on Vercel. Its serverless API routes talk to the Google Sheets & Drive APIs through a service account. Writes from staff go app → Sheets; the Apps Script you already have stays as the validation/automation engine and pushes changes back to the app via a webhook. Sign-in is Google. When you outgrow Sheets, swap the data layer for Postgres behind the same API — the front-end never notices.

Why this and not "rebuild everything in a database now": the Sheet is your team's mental model, your audit trail, and your fallback. Keeping it as the source of truth in phase 1 means the app is additive — if Vercel is down, the Sheet still works. You earn the right to a database by hitting real limits (concurrency, rows, query complexity), not by guessing.

02How "everything syncs" — concretely

Two directions, two mechanisms. Keep them separate and the system stays predictable.

DirectionMechanismNotes
App → Sheets
(staff edits a shift in the app)		API route writes the cell with the Sheets API service account, then lets your existing Apps Script validation run.Debounce + write in batches. Optimistic UI: show the change instantly, reconcile on response.
Sheets → App
(someone edits the Sheet directly)		Apps Script onEdit / installable trigger POSTs the change to /api/webhook/sheets with a shared secret.App updates its cache and pushes to open clients (poll every ~30s, or websockets/SSE later).
Read pathAPI reads from Vercel KV cache, refreshed on webhook or TTL (e.g. 60s).Sheets API has quotas — never read it on every page load.
The cache is not optional

Google Sheets API allows roughly 60 reads/min per user. With many employees opening the app, you'll blow past that instantly if each request hits Sheets. Put Vercel KV (Redis) in front: serve reads from cache, invalidate it on the webhook. This single decision is what makes the Sheets-as-backend approach scale to a whole team.

The webhook, in one snippet

// Apps Script — add to your installable onEdit, after the CODED mirror
function notifyApp_(range, oldV, newV) {
  try {
    UrlFetchApp.fetch(SCRIPT_PROPS.APP_WEBHOOK_URL, {
      method: 'post', contentType: 'application/json',
      headers: { 'X-Secret': SCRIPT_PROPS.SHARED_SECRET },
      payload: JSON.stringify({ a1: range.getA1Notation(), oldV, newV }),
      muteHttpExceptions: true
    });
  } catch (e) { Logger.log(e); }   // never block the edit
}

03Auth & the three roles

Use Google sign-in (NextAuth) so nobody manages another password. Map each signed-in email to a role; gate both UI and API by it.

RoleSeesCan do
EmployeeTheir own month, the current-week team view, their stats.Read; submit requests (swap, leave) — never write the grid directly.
ManagerEverything + the warnings & coverage dashboards.Edit shifts, approve requests, run validation, export.
OwnerAll of the above + settings.Manage members, thresholds, sharing, new-month roll.

This finally separates the two things the spreadsheet couldn't: employees get genuine self-service visibility without any risk to the grid, because their writes are requests a manager approves — not cell edits.

04Scaling — and when to leave Sheets

Sheets-as-backend is genuinely fine for one team for a long time. Move to a database when you cross one of these lines, not before:

Signal you've outgrown SheetsWhat it means
Multiple teams / locations / many months at onceYou want relational queries ("all nights for nurse X across Q2") Sheets can't do well.
Concurrent editors fighting over cellsYou need row-level writes + transactions, not a single grid.
> ~10k–50k rows, or slow reads even with cacheSheets API latency and the 10M-cell ceiling start to bite.
Real reporting / history / analyticsYou want SQL, not COUNTIF.
The migration is painless if you plan for it

Because every read/write already goes through your API layer, switching from a SheetsRepository to a PostgresRepository (Supabase or Neon — both have free tiers and great Vercel integration) is a back-end swap. Keep writing a one-way export back to Sheets so managers keep their familiar view and audit trail. The front-end code doesn't change at all.

05"Accessible for everyone" — what that actually requires

Mobile-first PWA

Most staff will check shifts on a phone. Ship an installable PWA: home-screen icon, offline cache of their schedule, fast cold start.

Greek, properly

Full el localisation (next-intl), Greek date/number formats, and the existing shift tokens as first-class labels.

WCAG 2.1 AA

Keyboard nav, focus states, semantic colour (the shift colours need text/icon backup for colour-blind users), 44px hit targets, screen-reader labels.

Low-friction entry

One-tap Google sign-in, magic-link fallback, no app-store install. Notifications (web push / email) for published schedules & approved requests.

Accessibility here is half a11y (the WCAG sense) and half reach (works on any phone, no install, in their language). Plan for both from the first screen.

Two things to get right early

Secrets: the service-account JSON and webhook secret live in Vercel env vars (and Apps Script Script Properties) — never in the repo. Quotas: design every read to come from cache; batch every write. Get these two right and the rest is ordinary app work.
